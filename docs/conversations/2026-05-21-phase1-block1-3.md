## 重要技術決策與發現

### exactOptionalPropertyTypes 與外部庫的衝突

`tsconfig.base.json` 開了 `exactOptionalPropertyTypes: true`，導致與 viem、Privy、Prisma 等庫的 type 不完全兼容。解決方式：

- `Web3Providers.tsx`：`defaultChain` 和 `supportedChains` 用 `as never` 繞過 viem Chain type 的 optional property 問題
- `PrismaUserRepository.ts`：update 時用 `...(input.field != null ? { field: input.field } : {})` 處理 undefined
- `JoseJwtService.ts`：用 `String()` / `Number()` 而非 non-null assertion

### @privy-io/node v0.18 API 變化

- `verifyAccessToken` 是 standalone function（不在 PrivyClient 上）
- `client.users()._get()` 是新的取用戶方式
- linked accounts 的 type 需要顯式標注

### OpenZeppelin v5 變化

- `__UUPSUpgradeable_init()` 已移除（v5 的 UUPSUpgradeable 不需要 init）
- rule 41 已更新反映此變化

## 產生的 ADR
