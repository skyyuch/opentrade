## 主要討論內容

### 1. 為何 CI 是 Phase 0 不能再延的最後一塊

開 session 時，當前進度回顧：

- Phase 0 commit #1 ~ number-nine 全部 ship：文件、monorepo、ui、db、api、web、
  console、contracts、infra/terraform 全部就緒。
- 但每次 commit 前還是靠 founder **人工**跑 `pnpm lint / typecheck /
format:check / test:unit` + `forge build/test/fmt --check` + `terraform
fmt -check / validate`。這 9 commits 沒出事是運氣 + 紀律。
- 進 Phase 1 之前必須把這條安全網織出來，否則：
  - 第一位外部 contributor 進來時，沒人能信他/她的 PR
  - 半夜推 main 時可能忘跑某個 gate（漏 typecheck 是最常見）
  - Renovate/Dependabot PR（commit number-ten 同時要設）需要獨立 CI 信號
    才能評估能不能 merge

→ 所以 Commit number-ten 一定要在 Phase 0 內 ship，不能延 Phase 1。

### 2. 為何要寫 ADR-0018 而不是直接寫 yaml

Agent 提案寫 ADR-0018，使用者確認「必要」。

關鍵：CI/CD 看似只是工程紀律，但實際上**綁了 6 個互相依賴的設計問題**，
每一個都會限制其他：

1. 一個 workflow 還是三個？
2. Node + pnpm 怎麼裝？（`pnpm/action-setup` vs Corepack）
3. Foundry 版本 pin 在哪？（workflow 還是 `foundry.toml`？）
4. Terraform CI 怎麼跑而不需要 AWS 認證？（鴿子蛋難題）
5. 哪些 gate Phase 0 要、哪些延 Phase 1+？
6. 用 Dependabot 還是 Renovate？怎麼編碼 Prisma 6 / Next 14 ignore 規則？

如果不寫 ADR，6 個月後新人問「為什麼三個 workflow 不合一個」、「為什麼
ignore Next 15」、「Terraform 為什麼沒 `plan`」，沒人答得出。

ADR-0018 為這 6 個問題寫了 11 個 coordinated decisions（D1-D11）+ 7 個
alternatives considered + 全 implementation notes 列 Phase 1+/4+ follow-ups。

### 3. 三 workflow file 拆分，不是一個 mega ci.yml

ADR-0018 D1 的核心決策。理由四條：

1. **Runner shape 異質**：TS 要 Node + pnpm + Corepack；Foundry 要
   `foundry-toolchain` + git submodule recursive；Terraform 要
   `setup-terraform`。三套 setup step 正交，硬塞一個 workflow 就是
   least-common-denominator。
2. **Path filter 異質**：`ci.yml` 必跑每個 PR（任何 package.json 動到
   都可能影響別包 typecheck）；`contracts.yml` 只跑 `packages/contracts/**`
   有變動的 PR（省 ~3 min/PR）；`terraform.yml` 只跑 `infra/terraform/**`
   變動的 PR。合一個 workflow 就喪失這個 granularity。
3. **Concurrency cancellation per-workflow**：一個 mega workflow 在 fast
   `git push --force-with-lease` 時會把 still-relevant 的 Foundry job
   跟 no-longer-relevant 的 TS job 一起 cancel。拆開就不會。
4. **Phase 4+ OIDC scope 窄**：Phase 4+ ADR-0019 要給 `terraform.yml`
   AWS OIDC role 才能跑 `terraform plan`。拆開後這個權限授予只波及
   一個 workflow，不是整個 repo 所有 CI job。

### 4. Terraform CI 的鴿子蛋：怎麼 validate 而不要 AWS 認證

這是 ADR-0018 D7+D8 的核心。問題：

- `bootstrap/state-backend/` workspace 是 local state，CI 上跑完
  state 沒地方收，且 chicken-and-egg。
- `environments/dev/` workspace 是 remote state 在 S3，要 AWS 認證
  才能 `init`。但 Phase 0 + rule 80 + ADR-0017 D11 明寫**不給 CI
  AWS 認證**，OIDC 留 Phase 4+ ADR-0019。

解法：`terraform init -backend=false -input=false` + `terraform validate`。

- `-backend=false`：告訴 Terraform「不要 init backend」。只下載
  provider plugin（從 `.terraform.lock.hcl` resolve；offline-safe）。
- `validate`：檢查 HCL 語法、module call 簽章、variable reference、
  provider schema。不碰 state、不碰 AWS。

代價：抓不到 cost 回歸（要 `plan` + 真實 state）、IAM evaluation
issue（要 `apply` against AWS）、resource attribute conflict（要既有
state）。這三類 gap Phase 4+ 補上。

→ 副效應：必須**commit `.terraform.lock.hcl`**，否則 CI `init` 會
重新從 registry resolve provider hash，每 PR 都不一樣。這就觸發了
todo `t2`。

### 5. `.terraform.lock.hcl` 從 .gitignore 解禁

當前 `.gitignore` 把 `.terraform.lock.hcl` 排掉了（沿用 Terraform
官方早期 template 預設）。但官方近年改口：**lock file 應該 commit**，
這是 reproducible build 的基礎。

ADR-0018 D7 把 lock file commit 列為**前置條件**。執行步驟：

```bash
# 在每個 workspace 跑（兩個都跑）
terraform providers lock \
  -platform=linux_amd64 \   # CI runner
  -platform=darwin_arm64 \  # founder Mac M-series
  -platform=darwin_amd64    # contributor Intel Mac (forward-looking)
```

每 provider 結果有 3 個 `h1:` hash，CI Linux runner 不對外連網就能
resolve。

同步把這條紀律寫進 rule 81 新章節「Provider lock file 紀律」+ 紅線
兩條（不可 `.gitignore` 排除、不可 bump `versions.tf` 不 refresh
lock）+ `infra/terraform/README.md` 加「Updating the provider lock
file」段落。

### 6. ESLint 阻擋 frontend runtime import `@opentrade/db`

rule 10 早就規定「frontend 不可 import packages/db」，但只有 code
review 在守。Commit #4 conversation 也提到「未來加 type-aware lint
rule」。Commit number-ten 是時候。

ADR-0018 D11 列為 implementation note。實作 in todo `t7`：

```js
// root eslint.config.mjs
{
  files: ['apps/web/src/**/*.{ts,tsx}', 'apps/console/src/**/*.{ts,tsx}'],
  rules: {
    '@typescript-eslint/no-restricted-imports': [
      'error',
      {
        patterns: [
          {
            group: ['@opentrade/db', '@opentrade/db/*'],
            message: '...rule 10 + ADR-0006...',
            allowTypeImports: true,  // ← rule 10 type-only exception
          },
          {
            group: ['@prisma/client', '@prisma/client/*'],
            message: '...rule 10 + ADR-0014 + rule 50...',
            allowTypeImports: true,
          },
        ],
      },
    ],
  },
}
```

**Smoke-test pattern**（建議未來 lint 規則新增都比照）：

1. 在 `apps/web/src/lint-smoke.test-fixture.ts` 寫 4 個 import：
   - `import type { User } from '@opentrade/db'` → 預期 allow
   - `import type { Prisma } from '@prisma/client'` → 預期 allow
   - `import { prisma } from '@opentrade/db'` → 預期 error
   - `import { PrismaClient } from '@prisma/client'` → 預期 error
2. 跑 `pnpm --filter @opentrade/web lint`，確認上面 4 個預期。
3. 刪 fixture file（必須在 commit 前刪掉）。
4. 重跑 `pnpm turbo run lint --filter='!@opentrade/contracts' --force`
   確認既有源碼仍 0 error。

這個 pattern 比「寫完 yaml 直接 push 看 CI」快 30 倍，且 PR 不會
留下廢檔。

### 7. Dependabot 不是 Renovate

ADR-0018 D9。Phase 0 取 Dependabot 因：

- GitHub-native，零額外 App install
- 對 9-package monorepo 的 grouping 已足
- ignore rule 直接編碼 ADR-0013 (Prisma 6) + docs/03-status open
  question (Next 14→15)

Renovate 的 power（`packageRules` 引擎、provider 級別控制）對 Phase
0 規模 overkill；當 contributor base 真實長到 5 人+ 再評估。

**Phase 0 Dependabot ignore 清單**（每條都 ref 對應 ADR / open question）：

| Package                            | Reason for ignore-major |
| ---------------------------------- | ----------------------- |
| `prisma`, `@prisma/client`         | ADR-0013 6.x pin        |
| `next`, `eslint-config-next`       | docs/03-status open Q   |

| `@types/react`, `@types/react-dom` | same                    |
| `storybook`                        | Phase 1+ 設計系統評估   |

不在 ignore 的：所有 minor / patch 都接收（透過 group 包成 batched PR
給 founder 一次過 review）。

### 8. Branch protection 為什麼是 UI follow-up

ADR-0018 D10。GitHub branch protection 設定**沒有 yaml-encodable
管道** —— 唯一 API 是 protected-branches REST API，要 admin PAT。
給 CI 一個 admin PAT 為了設「禁止 force push」是 bootstrap paradox
（要先信任這個 token 才能用它禁止濫用）。

解：把完整設定清單寫進 ADR-0018 D10 的表格，第一輪 PR 綠後 founder
在 GitHub Settings UI 對表設定。表格內容包含：

- Branch pattern: `main`
- Required status checks: `ci / lint`, `ci / typecheck`, `ci /
format`, `ci / test` + 條件性 `contracts / forge`, `terraform
/ validate`
- Require linear history（= squash merge per rule 70）
- Restrict force push（rule 70 紅線）
- Restrict deletions

第二位 contributor 加入時把「Required approving reviews」從 0 翻
1（同 PR 改 CODEOWNERS）。

### 9. gh CLI 裝法選擇：~/.local/bin tarball，不 Homebrew

執行 t10 開 PR 時發現 `gh` 沒裝。Agent 列三選項：

1. 直接下載 GitHub Release tarball 解壓到 `~/.local/bin/`（與
   Terraform 同 pattern；無 sudo、無 brew、零污染）
2. 先裝 Homebrew 再 `brew install gh`（修改 `/opt/homebrew` +
   `~/.zshrc`；長期 `brew update` 跑 200+ formula metadata 是
   隱形成本）
3. 使用者自己用 GitHub web 開 PR，回報結果給 agent

Agent 建議 #1，使用者採納。執行：

```bash
curl -L -o /tmp/gh.zip \
  'https://github.com/cli/cli/releases/download/v2.92.0/gh_2.92.0_macOS_arm64.zip'
unzip /tmp/gh.zip -d /tmp
cp /tmp/gh_2.92.0_macOS_arm64/bin/gh ~/.local/bin/gh
chmod +x ~/.local/bin/gh
gh --version  # → gh version 2.92.0
gh auth login  # 互動 web flow (user-driven)
```

→ 確立「**工具 footprint policy**」：開發者本機新工具一律先試
`~/.local/bin/` tarball，brew 只在 binary 不存在 / 需要 native deps
時才考慮。這呼應 Phase 0 的「最小依賴」精神。

### 寫新 CI workflow / 改現有 CI 時

- 所有 workflow 必含 `permissions: contents: read`（**先 deny**，
  Phase 4+ OIDC 才加 `id-token: write`）
- 所有 workflow 必含 `concurrency.group + cancel-in-progress: true`
- 所有 workflow 必含 `timeout-minutes`（防止 runaway 燒 CI minute）
- **絕不** 加 `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` 到 secrets
  （per rule 80 + ADR-0017 D11；OIDC 走 Phase 4+ ADR-0019）
- **絕不** 在 CI 跑 `terraform apply` 或 `terraform plan`（per
  rule 81 + ADR-0018 D7）
- 第三方 action 用 major-version tag（`@v4`, `@v3`），讓 Dependabot
  github-actions ecosystem 自動 PR；不用 commit SHA pin（Phase 0
  接受 supply-chain risk）

### 寫新 ESLint rule 時

- 用 smoke fixture pattern（per 上面 #6），不要寫完 push 看 CI
- Type-aware rule 用 `@typescript-eslint/no-restricted-imports` 而非
  ESLint 內建 `no-restricted-imports`（前者支援 `allowTypeImports`）
- 規則只 scope 到必要 file pattern（用 `files: [...]` overlay）；
  不要全 monorepo 一視同仁，避免誤傷

### 改 Terraform `versions.tf` 時

- **必同步**跑 `terraform providers lock -platform=linux_amd64
-platform=darwin_arm64 -platform=darwin_amd64`（per rule 81 +
  ADR-0018）
- 把 lock file delta 跟 `versions.tf` bump **同一 commit** 提
- 不在同一 commit 的話 CI Linux runner 會在 `init` 重新 resolve，
  然後失敗或 fingerprint drift

### 維護 ADR-0018 D11 deferral table

每完成一個 deferred item（Snyk、Slither、E2E、coverage、bundle
size 等），在它的 land PR 同時：

1. 更新 ADR-0018 D11 表格那一行（status → ✅ landed in #PR-XXX）
2. 更新 rule 70 「CI 必跑檢查」清單對應條
3. 更新 `docs/03-status.md` 反映 CI surface 變化

不更新會讓 deferral table 變成「永遠列在那裡沒人看的待辦清單」。
